# Domain Pitfalls

**Domain:** Visual polish additions to a high-performance event website
**Project:** MK Ultra Gravel — v8.0 visual polish milestone
**Researched:** 2026-03-31
**Confidence:** HIGH (primary sources: web.dev official docs, MDN, direct codebase inspection)

---

## Context

This file covers pitfalls specific to v8.0: adding animated backgrounds, a horizontal masonry gallery,
decorative SVG section dividers, tone images between sections, 19 new route photos, and a GPX
replacement. The site currently scores Lighthouse mobile 96, TBT 0ms, LCP 2.48s, CLS 0.054.

**The goal is zero regressions.** Every pitfall below has a "which phase" annotation so the roadmap
can front-load risk mitigation.

The existing z-index stack (confirmed by codebase inspection):
- `.grain-overlay`: `z-index: 9999` (fixed, `aria-hidden`, body level)
- `.escher-overlay`: `z-index: 9998` (fixed, `aria-hidden`, body level, `will-change: transform`)
- `.site-nav`: `z-index: 10000` (fixed)
- Section tone images: `position: absolute`, no z-index set (flow within `position: relative` sections)
- Section content: `position: relative; z-index: 10` (confirmed in index.astro)

### Pitfall 1: Masonry Gallery Without Width/Height Attributes Destroys CLS

**What goes wrong:**
The current PhotoGallery.astro renders `<img>` tags with no `width` or `height` attributes — only
CSS `aspect-[3/4]` forcing the container. Moving to a horizontal masonry layout means each row's
height is determined by the images themselves. If masonry uses absolute positioning (every JS-based
masonry library does), the gallery container has zero height on first paint. As thumbnails load,
the container expands, shifting everything below it. A gallery of 74 images without dimensions can
cause CLS of 0.3+ even with lazy loading.

The existing `photos.json` already stores `width` and `height` for every photo (generated by
`generate-thumbnails.js` writing sharp metadata back into the JSON). This data is available but
currently unused in the `<img>` tags.

**Why it happens:**
Developers add `loading="lazy"` and `decoding="async"` to every image assuming this is sufficient.
It prevents the browser from blocking on image load but does NOT prevent layout shift unless the
browser knows the dimensions in advance. CSS `aspect-ratio` on the container helps but only if the
container's width is already known — which it is not until the masonry algorithm runs.

**Warning signs:**
- CLS score jumps from 0.054 to >0.1 in Lighthouse after gallery change
- Images "snap" into position visibly during scroll past the gallery section
- Chrome DevTools CLS overlay highlights the gallery section in red

**Prevention:**
1. Pass `width` and `height` attributes from `photos.json` to every `<img>` in the gallery:
   ```astro
   <img
     src={thumbSrc}
     width={photo.width}
     height={photo.height}
     ...
   />
   ```
   Note: `photo.width`/`photo.height` are the FULL-SIZE dimensions from `generate-thumbnails.js`.
   The thumbnail is 400px wide with proportional height. Pass the full-size values — the browser
   uses them only to compute aspect ratio, not actual render dimensions.

2. For horizontal masonry with a fixed row height, set an explicit `height` on the masonry
   container before any images load. Fixed-height rows (all images scaled to same height, widths
   vary) prevent the container itself from causing CLS.

3. Avoid JS-based masonry libraries (Masonry.js, Isotope, etc.) entirely. They require a layout
   recalculation pass after DOM insertion, which fires AFTER first paint. This is the mechanism
   that causes CLS. Use CSS-only approach: `display: flex; flex-wrap: wrap; align-items: stretch`
   with images set to a fixed height and `width: auto`. This achieves the horizontal masonry look
   (variable-width images filling rows) with zero JS and zero layout shift.

**Phase:** Gallery replacement phase (whichever implements horizontal masonry).
**Confidence:** HIGH — confirmed by web.dev/optimize-cls official documentation and direct inspection of PhotoGallery.astro.

### Pitfall 2: JS Masonry Library Destroys TBT (Currently 0ms)

**What goes wrong:**
Any masonry library that runs JavaScript to calculate and apply absolute positioning adds main-thread
work during page load. Even a fast library (Masonry.js, Macy.js) runs synchronously or on
DOMContentLoaded and blocks the main thread for 50-200ms depending on item count. With 74 photos,
expect 100-300ms of TBT added — a catastrophic regression from the current 0ms.

**Why it happens:**
JS masonry works by:
1. Inserting all items into the DOM
2. Waiting for layout to complete (forces reflow)
3. Measuring each item's height
4. Applying `position: absolute; top: X; left: Y` to every item

Steps 2-4 are synchronous main-thread work. Each forced reflow during layout measurement can
individually exceed the 50ms long-task threshold.

**Warning signs:**
- Lighthouse TBT score moves from 0ms to any non-zero value
- Chrome DevTools Performance panel shows long "Layout" tasks during page load
- "Eliminate render-blocking resources" or "Reduce JavaScript execution time" appear in Lighthouse

**Prevention:**
Do not use JS masonry libraries. The horizontal masonry look (fixed-height rows, variable-width
images) is achievable with pure CSS flexbox. The gallery section already uses Tailwind utilities —
this approach fits the existing pattern.

CSS-only approach:
```css
.masonry-row {
  display: flex;
  gap: 4px;
  overflow-x: auto;
  height: 300px; /* fixed row height */
}
.masonry-row img {
  height: 100%;
  width: auto;
  flex-shrink: 0;
  object-fit: cover;
}
```
Rows are pre-calculated at build time (Astro server render), not in the browser. Zero JS.

**Phase:** Gallery replacement phase.
**Confidence:** HIGH — web.dev animations guide confirms JS layout recalculation causes TBT regressions.

### Pitfall 3: GPX Replacement Cascades Silently Through Five Downstream Consumers

**What goes wrong:**
The GPX file (`MK_Ultra.gpx`) is consumed by `parse-gpx.js`, which produces `route-data.json`. This
then feeds four downstream scripts: `resolve-annotations.js` (sector/KOM mile positions),
`match-photos.js` (photo lat/lon mapping), the Leaflet route map (direct `public/mk-ultra.gpx` copy),
and the ElevationProfile chart (route-data.json). A new GPX file changes total mileage, trackpoint
count, and intermediate coordinates.

The critical silent failure mode: if new GPX mileage is ~100mi but old photo mile markers in
`photo-manifest.js` still reference positions on the OLD route geometry, EXIF-less photos get mapped
to wrong coordinates. The pipeline runs without errors (just clamping warnings), and the map shows
photos in incorrect locations. This can pass visual review if the reviewer isn't cross-checking
every photo marker.

Additionally, `resolve-annotations.js` maps sector/KOM annotations by mile marker. If the new
GPX redistributes distance (e.g., GPS smoothing changes cumulative calculations), a sector listed
at `startMi: 45.2` may now point to a different physical location on the new route.

**Why it happens:**
The GPX replacement looks like a simple file swap. The pipeline is designed to be idempotent and
runs without human review gates. Warnings about mile markers exceeding route end are logged to
console but do not halt the build — they are easy to miss in CI output.

**Warning signs:**
- `match-photos.js` outputs "WARNING: Mile marker X exceeds route end" — any of these is a red flag
- Map photo markers cluster near the route endpoint (clamping behavior)
- Total mileage shown on the site changes unexpectedly
- `route-data.json`'s `meta.totalMi` is different from the expected ~100 miles

**Prevention:**
1. Run the full pipeline in isolation after GPX swap, before committing other changes
2. Compare `route-data.json` `meta` output (totalMi, elevationGainFt, trackpoints) against expected
   values from the Strava source export
3. Check for ANY clamping warnings in `match-photos.js` output — if new GPX is longer, previously
   valid mile markers should all still be valid; warnings indicate a regression
4. Visually verify at least 3 map photo markers span the route correctly (start, middle, end)
5. Add the GPX swap as its own isolated phase/task before any other v8.0 changes

**Phase:** GPX replacement phase — must run first, before any other visual changes, to establish
a clean data baseline.
**Confidence:** HIGH — confirmed by direct inspection of the pipeline scripts and the cascading
dependency chain from `MK_Ultra.gpx` → `route-data.json` → `annotations.json` + `photos.json`.

### Pitfall 4: New Tone Images Placed Incorrectly Become LCP Candidates

**What goes wrong:**
The existing tone images use `loading="lazy"` for all below-fold images and `loading="eager"` with
`fetchpriority="high"` for the hero tone image (`CIA-MKULTRA-IG_Page_01.webp`). This is correct.

New tone images placed between sections (added in v8.0) could become LCP candidates if:
- They are placed at or near the top of the page (above fold on some viewport heights)
- They are given `loading="eager"` without being the actual LCP resource
- They are larger files than the existing ~10-60KB tone WebPs

If a new tone image is preloaded (via `<link rel="preload">`) but is not the actual LCP element,
Chrome penalizes the page for an unused preload and delays other resources.

**Why it happens:**
Copy-paste of the hero section's eager/fetchpriority pattern onto below-fold images is a common
mistake. "More eager loading = faster" is not true below the fold.

**Warning signs:**
- Lighthouse flags "Largest Contentful Paint image was lazily loaded" — means a non-lazy image
  near the top is being measured as LCP unexpectedly
- LCP increases beyond 2.48s baseline
- DevTools Network panel shows tone images loading before Leaflet/Chart.js bundles

**Prevention:**
1. All new tone images that are NOT in the first viewport: use `loading="lazy"`, no `fetchpriority`
2. Follow the existing pattern: only `/tone/CIA-MKULTRA-IG_Page_01.webp` gets eager + preload
3. Run `convert-tone-images.js` (or equivalent) for every new tone image — the script compresses
   aggressively (q45-q60, 350-1000px wide) because 12% opacity makes compression artifacts invisible
4. Add new source images to the `TONE_IMAGES` array in `convert-tone-images.js` before using them

**Phase:** Tone image integration phase.
**Confidence:** HIGH — confirmed by web.dev lcp-lazy-loading article and direct inspection of
`index.astro` (existing tone image pattern) and `convert-tone-images.js`.

### Pitfall 5: Animated Background Creates Too Many Compositor Layers on Mobile

**What goes wrong:**
The existing `escher-overlay` already has `will-change: transform` (confirmed in global.css line 102),
promoting it to its own compositor layer. The `grain-overlay` is static (no `will-change`). Adding
an animated lizard background with `will-change: transform` or `animation` creates a third
compositor layer active during scroll.

Mobile GPUs have significantly less VRAM than desktop. Chrome's compositor has a per-page layer
budget. Exceeding it causes the browser to "squash" layers back together, which triggers a full
repaint. On low-end Android devices this manifests as jank during scroll, not a crash. On older
iPhones it can cause the tab to reload.

**Why it happens:**
CodePen demos are built for desktop Chrome where layer budgets are generous. The author applies
`will-change` liberally because it "makes animations smooth." Mobile behavior is not tested.

**Warning signs:**
- Chrome DevTools Layers panel shows >4 active layers simultaneously
- Scroll performance degrades on mobile when the animated section is visible
- "Layer count" metric in DevTools is unusually high
- iOS Safari tab reloads when scrolling to the animated section

**Prevention:**
1. Do NOT add `will-change: transform` to the lizard background element — let the browser decide
   when to promote it
2. Use `animation` with only `transform` and/or `opacity` — these are compositor-safe but the
   browser promotes the layer only when needed, not permanently
3. Gate the animation behind `@media (prefers-reduced-motion: no-preference)` and test with
   reduced motion ON to verify the background is still presentable as a static image
4. Test on a throttled (4x slowdown CPU) Chrome DevTools profile before shipping
5. The animation should be decorative-only — if it doesn't run, the layout must still be correct

**Phase:** Animated background phase.
**Confidence:** HIGH — confirmed by web.dev animations guide (will-change as last resort, layer
creation has costs) and Motion.dev magazine (mobile GPU memory exhaustion documented concern).

### Pitfall 6: CodePen Adaptation Assumes Global Scope, Viewport Dimensions, and Full-Page Context

**What goes wrong:**
CodePen demos run in a sandboxed `<iframe>` with a reset CSS environment. Common adaptation issues
when extracting CSS-only animations:

1. **Viewport units**: CodePen pens that use `100vw`/`100vh` assume the pen fills the viewport.
   Embedded in a page section, `100vw` includes the scrollbar and causes horizontal overflow.
   `100vh` in a section context may be wrong by the height of the nav (48px `pt-12` on body).

2. **CSS variable conflicts**: Pens that define `--color`, `--size`, etc. at `:root` will conflict
   with the existing Tailwind theme tokens. The site uses `--color-*` prefixed tokens at `:root`
   via `@theme`. A CodePen variable `--color: red` would be unset but could shadow a Tailwind variable
   depending on specificity.

3. **Inline styles from JS**: Some CodePen animations require a small JS snippet to set CSS custom
   properties (e.g., mouse position for parallax). Inline `<style>` tags and global JS that modifies
   `document.documentElement.style` conflict with Tailwind v4's CSS-first config approach.

4. **`:root` background assumptions**: The lizard background and topographic dividers are likely
   designed against a white or neutral background. The site background is `oklch(0.10 0.01 250)` —
   near-black. SVG fill colors and CSS gradients from a light-background pen will not be visible
   without inversion or color adjustment.

**Why it happens:**
CodePen pens are self-contained demos. Authors don't test in dark-background, scrollable-page
contexts with existing global CSS.

**Warning signs:**
- Horizontal scrollbar appears after adding the animation
- Animation colors are invisible or show as solid blocks on the dark background
- CSS custom property values in the animation appear wrong (shadowed by pen's `:root` declarations)
- Animation behaves differently at different scroll positions vs. in the CodePen preview

**Prevention:**
1. Scope all extracted CSS inside the component's selector (e.g., `.lizard-bg { ... }`) to prevent
   `:root` variable leakage
2. Replace `100vw` with `100%` for width; replace `100vh` with `min-height: 100%` scoped to the
   section container
3. Adjust fill/stroke colors from the pen's assumed background to the site's dark palette before
   deploying (do not assume CodePen colors will be visible)
4. Remove any `will-change` the pen adds globally — audit each property and re-add only if needed
5. Test with scrolled context: the pen preview starts at the top of the pen, but the section may
   be 1500px below the page top

**Phase:** Animated background phase and SVG divider phase.
**Confidence:** MEDIUM — based on pattern analysis of CodePen adaptation issues; no single
authoritative source, but consistent across community discussions.

### Pitfall 7: mix-blend-mode on Tone Images Creates Unintended Stacking Context

**What goes wrong:**
The existing `.tone-image` class uses `mix-blend-mode: lighten`. MDN confirms that `mix-blend-mode`
creates a new stacking context on the element. This means any child elements inside a
`position: relative` section that uses a tone image as an absolute background may have their
z-index behavior affected.

Specifically: the new tone images being placed "inside cards" (GravelSectors, KomSegments) could
interfere with card hover states, PhotoSwipe lightbox overlays, or the Leaflet map popup z-indexing
if the tone image's stacking context is not contained properly.

Additionally, `mix-blend-mode: lighten` on a fixed overlay does not blend against the fixed
`grain-overlay` or `escher-overlay` the way you might expect — it blends against the element
BELOW it in the DOM, not against what is visually below it on screen. This can produce unexpected
color shifts where tone images overlap with the Leaflet map tiles (which are dark CARTO tiles).

**Why it happens:**
Blend modes feel like a pure visual effect but create stacking contexts with real layout and
z-index consequences.

**Warning signs:**
- Card hover effects stop working when tone image is added as card background
- PhotoSwipe lightbox appears behind the card section instead of above everything
- Map area shows unexpected color wash where tone image bleeds over it

**Prevention:**
1. Add `isolation: isolate` to the section containers that have tone image backgrounds, to contain
   the blend mode scope
2. Keep tone images as `position: absolute` inside their section (current pattern) — do not apply
   them as direct backgrounds on the cards themselves
3. If tone images are placed inside cards, set `isolation: isolate` on the card container and
   verify PhotoSwipe still opens correctly (test by clicking a gallery item after adding card tones)
4. Test with the Leaflet map visible — scroll to the Route section and confirm map tiles are not
   color-shifted

**Phase:** Tone image integration phase.
**Confidence:** HIGH — confirmed by MDN `mix-blend-mode` formal definition ("Creates stacking context: yes") and direct inspection of the existing tone image pattern.

### Pitfall 8: 19 Additional Photos Increase Gallery Page Weight Without Thumbnail Audit

**What goes wrong:**
The photo pipeline generates 400px WebP thumbnails at q80 (~15-30KB each). Adding 19 photos adds
285-570KB of lazy-loaded thumbnail payload. Individually each image is fine, but:

1. If any new photos are portrait-mode shots at 1536x2048 (the dominant format in the manifest),
   their thumbnails will be 400x533px — larger than the horizontal masonry row height needs. The
   horizontal masonry layout at a ~300px row height only needs ~225px-wide thumbnails for portrait
   images. Oversized thumbnails waste bandwidth.

2. The assign-card-photos.js algorithm picks cover photos based on mile range proximity. Adding
   19 new photos may change which photo is selected as cover for each sector/KOM card, requiring
   visual review of all card images after the pipeline runs.

3. PhotoSwipe full-size lightbox still loads the original 1536x2048 JPEG (~1.5MB) when a user
   clicks a thumbnail. With 74 photos, this is fine (each is a user-initiated action). The pitfall
   is accidentally setting `fetchpriority="high"` on lightbox sources in the gallery markup, which
   would cause ALL full-size images to preload simultaneously.

**Why it happens:**
Photo addition feels like a content update, not a performance change. The pipeline automation hides
the size implications. Card cover photo changes are not obvious until visual inspection.

**Warning signs:**
- Total transfer size increases by more than 600KB over baseline after photo addition
- Any sector/KOM card shows a visually poor cover photo that was previously good
- Lighthouse "Properly size images" warning appears for gallery thumbnails

**Prevention:**
1. After running the pipeline with new photos, diff `annotations.json` to see which coverPhoto
   fields changed — visually check each changed card
2. Audit thumbnail sizes: if horizontal masonry uses a ~300px row height, consider reducing
   thumbnail width from 400px to 300px in `generate-thumbnails.js`. 300px at q80 produces ~8-18KB
   per thumbnail — a 40% reduction per image
3. Never add `fetchpriority` or `preload` to the full-size gallery image sources
4. Run `npm run build` and check the `dist/_astro` output size delta before and after photo addition

**Phase:** Photo addition phase (adding 19 new photos).
**Confidence:** HIGH — based on direct inspection of `generate-thumbnails.js` (400px/q80),
`assign-card-photos.js` (mid-range proximity algorithm), and PhotoGallery.astro (no fetchpriority on lightbox sources — confirm this is maintained).

### Pitfall 9: SVG Topographic Dividers Without aria-hidden Break Screen Reader Flow

**What goes wrong:**
Decorative SVG dividers between sections have complex path data. If rendered inline without
`aria-hidden="true"`, screen readers will attempt to read them, potentially reading dozens of
coordinate numbers as text. This does not break accessibility functionally but creates a deeply
confusing experience.

**Prevention:**
All decorative SVGs (dividers, background patterns) must have `aria-hidden="true"` and
`focusable="false"`. This matches the existing pattern: `escher-overlay`, grain, and the hero
Penrose triangle SVG all have `aria-hidden="true"`.

**Phase:** SVG divider phase.
**Confidence:** HIGH — confirmed by WCAG decorative image guidance and the existing site pattern.

### Pitfall 10: Reduced-Motion Gate Missing on New Animations Breaks WCAG 2.3.3

**What goes wrong:**
The existing site gates all animations with `prefers-reduced-motion`. The scroll-reveal observer
in `index.astro` explicitly checks `window.matchMedia("(prefers-reduced-motion: reduce)").matches`
and skips entirely if true. The `glitch` animation in index.astro uses clip rect animation which
is NOT gated — this is a pre-existing issue.

New animations (lizard background, any CSS keyframe animation) that run continuously without a
`prefers-reduced-motion: reduce` gate violate WCAG 2.3.3. The W3C requirement is that animations
triggered by interaction can be disabled, and WCAG 2.3.1 covers flashing content. Continuous
background animations lasting >5 seconds require a pause mechanism even for users who have NOT
enabled reduced motion.

**Prevention:**
1. All new `@keyframes` animations must be wrapped in:
   ```css
   @media (prefers-reduced-motion: no-preference) {
     .animated-element {
       animation: ...;
     }
   }
   ```
2. For animations lasting >5 seconds (the lizard background will run indefinitely), provide a
   visual pause button or ensure the animation is imperceptible enough that WCAG 2.3.1 flashing
   thresholds are not met
3. The "subtle" requirement in the milestone context helps here — if the animation runs at <3
   cycles/second and does not flash, it is below the photosensitive seizure threshold

**Phase:** Animated background phase.
**Confidence:** HIGH — WCAG 2.3.3 official W3C documentation and the existing site's established
reduced-motion pattern.

### Pitfall 11: Horizontal Gallery Scroll Container Captures Touch Events on Mobile

**What goes wrong:**
A horizontally scrollable gallery container inside a vertically scrolling page creates a scroll
event conflict on mobile. iOS Safari and Android Chrome use heuristics to determine scroll
direction intent, but if the gallery container has `overflow-x: auto` or `overflow-x: scroll`,
touch events within the gallery hitbox are captured by the gallery, not the page. Users who
attempt to scroll the page by starting their swipe over the gallery will scroll the gallery
instead, creating a confusing and stuck-feeling UX.

This is a UX pitfall, not a performance one, but it can cause users to incorrectly experience
the site as broken.

**Prevention:**
1. Use a `touch-action: pan-y` CSS rule on the gallery container — this tells the browser to
   prefer vertical scrolling when the user intends to scroll vertically, only capturing
   horizontal swipes for the gallery
2. Alternatively, avoid `overflow-x: scroll` entirely — if the horizontal masonry is rendered
   as multiple rows (2-3 rows of images, each row fitting within viewport width), there is no
   horizontal scroll conflict

**Phase:** Gallery replacement phase.
**Confidence:** MEDIUM — standard mobile scroll conflict pattern; confirmed by MDN `touch-action`
documentation.

### Pitfall 12: `will-change` on `.escher-overlay` Already Occupies a Layer

**What goes wrong:**
The `.escher-overlay` has `will-change: transform` (global.css line 102). This permanently
occupies a compositor layer for the fixed background pattern. This was presumably added to prevent
the escher pattern from repainting during scroll.

If the new animated lizard background or topographic SVG dividers also use `will-change: transform`,
and if `will-change` is applied to multiple elements in different sections simultaneously, Chrome
may de-prioritize the escher layer, causing it to repaint (defeating the original purpose) or
cause layer compositing overhead that adds to scroll jank.

**Warning signs:**
- Chrome DevTools Layers panel shows >5 layers after changes
- Escher pattern visibly flickers during scroll after adding new animated elements

**Prevention:**
1. Audit `will-change` usage after all v8.0 features are added — target maximum 3 simultaneously
   active `will-change` layers
2. If the escher overlay flickers, consider removing `will-change: transform` from it (the overlay
   is static SVG — it only needs promotion if it was flickering without it, which may have been a
   pre-existing issue worth re-evaluating)
3. Never add `will-change` preemptively to decorative elements "just in case"

**Phase:** Final integration/polish review.
**Confidence:** MEDIUM — based on web.dev animation guide ("will-change causes layer creation,
which has memory and paint costs") and direct inspection of global.css.
