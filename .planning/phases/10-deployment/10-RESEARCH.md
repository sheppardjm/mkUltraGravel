# Phase 10: Deployment - Research

**Researched:** 2026-03-27
**Domain:** Cloudflare Pages deployment for Astro 6 static site
**Confidence:** HIGH

## Summary

This phase deploys a fully static Astro 6 site to Cloudflare Pages with git-based CI/CD. The site has a multi-step prebuild pipeline that uses sharp (native Node module) to process images and generate thumbnails, parses GPX data, and extracts EXIF metadata. The key deployment challenge is ensuring all source assets (50 route photos, GPX file) are available in the git repository so the build pipeline works on Cloudflare's Linux build image.

Cloudflare Pages v3 build image ships with Node.js 22.16.0 by default, which is compatible with this project's Node 22 requirement. No Astro adapter is needed for static sites -- Astro pre-renders all pages at build time and Cloudflare Pages serves the `dist/` output as static files. The `npm run build` command triggers the `prebuild` lifecycle hook automatically, so the existing data pipeline runs without modification on Cloudflare.

The critical prerequisite is committing source assets (the `images/` directory and `MK Ultra.gpx`) to git, and adding a proper `.gitignore` to exclude build artifacts (`dist/`, `node_modules/`, `.astro/`). Currently the project has NO `.gitignore` file.

**Primary recommendation:** Commit source images and GPX to git, add .gitignore, create Cloudflare Pages project via dashboard with `npm run build` as build command and `dist` as output directory -- no adapter, no wrangler config needed.

## Architecture Patterns

### Build Pipeline Flow on Cloudflare Pages

```
git push to main
    |
    v
CF Pages clones repo
    |
    v
npm install            (installs sharp linux-x64 binaries, all deps)
    |
    v
npm run build          (npm auto-runs "prebuild" first)
    |
    +-- prebuild: node scripts/generate-data.js
    |       1. Copies images/ -> public/images/
    |       2. Parses GPX -> public/data/route-data.json + public/mk-ultra.gpx
    |       3. Resolves annotations -> public/data/annotations.json
    |       4. Matches photos (EXIF) -> public/data/photos.json
    |       5. Generates WebP thumbnails (sharp) -> public/images/thumbs/
    |       6. Converts hero image to WebP (sharp)
    |       7. Converts tone images to WebP (sharp)
    |
    +-- build: astro build -> dist/
    |
    v
CF Pages serves dist/ on CDN
```

### Anti-Patterns to Avoid
- **Installing @astrojs/cloudflare adapter for static sites:** The adapter is for SSR/hybrid mode only. A fully static Astro site needs no adapter -- it generates plain HTML/CSS/JS.
- **Using `astro build` directly as build command:** Use `npm run build` instead, which triggers the `prebuild` hook automatically. Calling `astro build` directly skips the prebuild step.
- **Using Wrangler for static Pages deploy:** Wrangler is for Workers/SSR. For git-integrated Pages with a static site, dashboard setup is sufficient.
- **Setting NODE_VERSION to an exact minor version that differs from CF default:** Use just `22` in `.node-version` to get the latest Node 22.x on the build image. Specifying `22.22.2` (Volta pin) may not be available on CF.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTPS certificates | Manual cert management | Cloudflare Pages automatic SSL | CF auto-provisions and renews certs for custom domains |
| CDN/caching | Custom cache headers | CF Pages built-in caching | CF Pages has optimized defaults; fingerprinted `_astro/` assets already get long cache |
| CI/CD pipeline | GitHub Actions workflow | CF Pages git integration | Dashboard connect-and-forget; rebuilds on every push to main |
| Image optimization at edge | CF Image Resizing | sharp in prebuild | Images are already processed at build time; no runtime processing needed |
| Redirect rules | Server-side routing | `_redirects` file (if needed) | CF Pages parses `_redirects` from output dir |

**Key insight:** For a static site, Cloudflare Pages handles virtually all ops concerns (SSL, CDN, CI/CD, DDoS) out of the box. The deployment task is almost entirely "connect repo and configure build."

## Common Pitfalls

### Pitfall 1: Source Images Not in Git
**What goes wrong:** Build fails because `scripts/generate-data.js` tries to read from `images/` directory, which doesn't exist in the cloned repo.
**Why it happens:** The 50 route photos in `images/` (33MB) are currently NOT tracked in git. The prebuild step copies them to `public/images/` and reads EXIF data from them.
**How to avoid:** `git add images/` and `git add "MK Ultra.gpx"` before first deploy. Both are required by the build pipeline.
**Warning signs:** Build log shows "ENOENT: no such file or directory" or "images/ not found."

### Pitfall 2: Missing .gitignore Causes Bloated Repo
**What goes wrong:** `node_modules/` (173MB), `dist/` (30MB), `.astro/`, `.DS_Store` files get committed.
**Why it happens:** The project currently has NO `.gitignore` file.
**How to avoid:** Create `.gitignore` BEFORE committing source images. Include: `node_modules/`, `dist/`, `.astro/`, `public/images/` (generated), `public/images/thumbs/` (generated), `.DS_Store`.
**Warning signs:** `git status` shows hundreds of files in node_modules or dist.

### Pitfall 3: Using Wrong Build Command
**What goes wrong:** Data pipeline doesn't run; `dist/` is missing generated data/images.
**Why it happens:** Setting `astro build` as the build command instead of `npm run build` skips the npm `prebuild` lifecycle hook.
**How to avoid:** Always use `npm run build` as the CF Pages build command. npm automatically runs `prebuild` before `build`.
**Warning signs:** Site deploys but shows broken images, missing map data, or no photos.

### Pitfall 4: Sharp Fails on Build Image
**What goes wrong:** `npm install` fails to install sharp's native binaries for the linux-x64 platform.
**Why it happens:** The `overrides.vite=^7` in package.json or dependency resolution conflicts could cause issues. Also, sharp 0.34.x requires specific platform binaries.
**How to avoid:** The CF Pages v3 build image (Ubuntu 22.04, Node 22) should handle sharp installation natively. If it fails, set `SHARP_IGNORE_GLOBAL_LIBVIPS=1` as an environment variable. Alternatively, pre-commit the generated thumbnails to `public/images/thumbs/` and skip the sharp step.
**Warning signs:** Build log shows "sharp" installation errors or missing libvips.

### Pitfall 5: Node Version Mismatch
**What goes wrong:** Build fails with Astro 6 compatibility errors or rollup binary issues.
**Why it happens:** CF Pages defaults to Node 22.16.0 on v3, but if the project somehow gets assigned to v1/v2 build system, it may get Node 12 or Node 18.
**How to avoid:** Add `.node-version` file with content `22` AND set `NODE_VERSION=22` environment variable in CF Pages dashboard as belt-and-suspenders.
**Warning signs:** Build log shows old Node version or "unsupported engine" errors.

### Pitfall 6: public/images/ Committed to Git AND Generated by Prebuild
**What goes wrong:** Git tracks `public/images/` (26MB of generated files) causing repo bloat and merge conflicts.
**Why it happens:** The prebuild copies from `images/` to `public/images/` and generates thumbnails. If `public/images/` is in git, every regeneration creates diffs.
**How to avoid:** Add `public/images/` to `.gitignore`. Only commit `images/` (source). The prebuild regenerates `public/images/` and `public/images/thumbs/` on every build.
**Warning signs:** Large diffs in `public/images/` on every commit.

### Pitfall 7: Volta Not Available on CF Build Image
**What goes wrong:** If any script references `volta run`, it fails on CF.
**Why it happens:** Cloudflare Pages build image does not include Volta. The project uses Volta locally for macOS-specific Node version pinning.
**How to avoid:** The build scripts use bare `node` and `npm`, not `volta run`. The CF build image's Node 22 handles everything directly. No changes needed to scripts.
**Warning signs:** "volta: command not found" in build log.

## Open Questions

1. **Production domain name**
   - What we know: The roadmap says "custom domain" but no specific domain is specified anywhere in the project docs.
   - What's unclear: What domain (e.g., mkultragravel.com, mkultra.bike, etc.) the user wants to use.
   - Recommendation: The planner should note this as a user-provided input. The `*.pages.dev` subdomain works immediately; custom domain is an additive step.

2. **Cloudflare account**
   - What we know: Deployment requires a Cloudflare account (free tier is sufficient).
   - What's unclear: Whether the user already has a Cloudflare account and whether the domain's DNS is already on Cloudflare.
   - Recommendation: Plan should include account setup as a prerequisite check, not a step to create.

3. **GitHub repository**
   - What we know: CF Pages connects to GitHub (or GitLab). The project is a git repo.
   - What's unclear: Whether the repo is pushed to GitHub already and whether it's public or private.
   - Recommendation: Plan should verify remote exists and repo is pushed before CF Pages setup.

4. **Sharp build reliability on CF Pages**
   - What we know: sharp 0.34.x uses prebuilt platform-specific binaries. CF Pages v3 runs Ubuntu 22.04 with Node 22. sharp should install linux-x64 binaries via `npm install`.
   - What's unclear: No direct confirmation that sharp 0.34.x builds successfully on CF Pages v3 specifically.
   - Recommendation: Plan the first deploy as a test. If sharp fails, fallback is to pre-commit generated thumbnails and skip the sharp step on CF.
