## Summary

Phase 6 is a purely static rendering phase — no new libraries required. The work is: (1) load `annotations.json` at build time in Astro frontmatter, (2) iterate over sectors/KOM/restock arrays and render styled card markup, (3) apply the existing design system tokens and component classes. This is the simplest phase in the project architecturally.

The central technical decision is **how to load annotations.json at build time**. The file lives in `public/data/` (generated by the prebuild script), which makes it inaccessible via ES module `import`. The correct approach is `fs.readFileSync` in the Astro component frontmatter — this is valid because Astro frontmatter runs on Node.js at build time. Alternatively, the generate-data.js prebuild script could also write a copy of annotations.json to `src/data/`, enabling a direct `import`. Either path works; `fs.readFileSync` is simpler and avoids modifying the data pipeline.

The visual design follows the established brutalist pattern: Space Mono font, oklch palette, border/surface tokens, `.classified-border` and `.stamp` components. The Paris-Roubaix star rating for sectors should use filled/empty star characters (★/☆) with the same `starColors` scale already used by the map (`RouteMap.astro`) and elevation profile (`ElevationProfile.astro`).

**Primary recommendation:** Use `fs.readFileSync` + `JSON.parse` in the frontmatter of a new `GravelSectors.astro` component (and optionally `KomSegments.astro`, `RestockPoints.astro`). Render cards with Tailwind utility classes using the existing token set. No new dependencies required.

## Architecture Patterns

### Recommended Project Structure

```
src/
├── components/
│   ├── GravelSectors.astro    # Reads annotations.sectors, renders Paris-Roubaix cards
│   ├── KomSegments.astro      # Reads annotations.kom, renders KOM cards
│   └── RestockPoints.astro    # Reads annotations.restock, renders simple list/cards
├── pages/
│   └── index.astro            # Imports all three, places in #sectors section
└── styles/
    └── global.css             # Unchanged — all needed tokens already defined
```

Each annotation type gets its own component. This keeps `index.astro` clean and makes each card type individually testable. All three components can be imported and dropped into the existing `#sectors` section, with `#info` remaining as a placeholder for Phase 7.

### Pattern 1: Build-Time JSON Loading with fs.readFileSync

**What:** Read `public/data/annotations.json` in Astro frontmatter using Node's `fs` module. Valid because frontmatter runs server-side at build time.

**When to use:** When the JSON file is generated by a prebuild script into `public/data/` and cannot be `import`-ed via ES modules.

**Example:**
```astro
---
// GravelSectors.astro
// Source: Astro docs — frontmatter executes at build time on Node.js
import { readFileSync } from 'fs';
import { join } from 'path';

const annotationsPath = join(process.cwd(), 'public', 'data', 'annotations.json');
const annotations = JSON.parse(readFileSync(annotationsPath, 'utf-8'));
const sectors = annotations.sectors;
---
```

**Why not direct import:** `import annotations from '../../../public/data/annotations.json'` — Vite cannot import from `public/` because Astro excludes the `public/` directory from Vite's module graph. Attempting this throws a build error.

**Alternative (also valid):** Modify `generate-data.js` to also write `src/data/annotations.json`, then `import annotations from '../data/annotations.json'`. This is cleaner for TypeScript type inference but requires modifying the data pipeline.

### Pattern 2: Star Rating Rendering

**When to use:** Sector cards — each sector has a `stars` field (1–5).

**Example:**

### Anti-Patterns to Avoid

- **Fetching annotations.json in a client `<script>` tag:** Causes FOUC (flash of unstyled content) — cards appear empty then populate. This is a static content problem; render server-side.
- **Importing from `public/` via ES module `import`:** Vite excludes `public/` from its module graph. `import data from '../../public/data/annotations.json'` will throw a build error.
- **Using CSS `var()` references for the star color scale:** Decision 03-03 documents that `var()` refs may not resolve in some contexts. Use raw oklch values or the hex codes already in use on the map.
- **Adding a new `@theme` token for star colors:** The star color scale is defined inline in `RouteMap.astro` and `ElevationProfile.astro`. Duplicate it inline in the new components too — it's 5 hex values, not worth a new abstraction.
- **Rendering all three card types into a single component:** Harder to maintain and breaks the single-responsibility pattern established by `RouteMap.astro` and `ElevationProfile.astro`.

## Common Pitfalls

### Pitfall 1: Attempting to Import JSON from public/

**What goes wrong:** Developer writes `import annotations from '../../public/data/annotations.json'` in the frontmatter. Vite throws a module resolution error at build time.

**Why it happens:** Vite's module graph does not include the `public/` directory by design — those files are served as-is, not processed.

**How to avoid:** Use `fs.readFileSync(join(process.cwd(), 'public', 'data', 'annotations.json'), 'utf-8')` in frontmatter.

**Warning signs:** Build errors referencing "Cannot find module" or "file not found" for the public path.

### Pitfall 2: process.cwd() in Component vs. Page Context

**What goes wrong:** `process.cwd()` resolves relative to where the Astro build is invoked, not where the component file lives. In this project (run from project root), `process.cwd()` is the project root — which is correct.

**Why it happens:** Developers sometimes expect `__dirname` to give the component directory. In ESM modules (which Astro uses), `__dirname` is not available.

**How to avoid:** Use `process.cwd()` for paths relative to the project root. Use `import.meta.url` with `fileURLToPath` if you need the component file's directory. For this project: `join(process.cwd(), 'public', 'data', 'annotations.json')` is correct.

**Warning signs:** `ENOENT: no such file or directory` errors during `astro build` or `astro dev`.

### Pitfall 3: `__dirname` Not Available in Astro ESM Context

**What goes wrong:** Developer writes `const data = JSON.parse(fs.readFileSync(__dirname + '/../public/data/annotations.json'))`. `__dirname` is undefined in ESM modules, causing a ReferenceError.

**Why it happens:** Astro components are ESM modules, which don't have the CommonJS `__dirname` global.

**How to avoid:** Replace `__dirname` with `process.cwd()` (project root) or derive the path with `fileURLToPath(new URL('../public/data/annotations.json', import.meta.url))`.

**Warning signs:** `ReferenceError: __dirname is not defined`.

### Pitfall 4: Missing `relative z-10` on Inner Content Div

**What goes wrong:** Cards render behind the `tone-image` overlay (which is `absolute` positioned). New content appears invisible or very faint.

**Why it happens:** The section pattern requires `<div class="relative z-10">` wrapping the actual content to stack above the background image. If the new component's root element lacks this, it falls behind the tone image.

**How to avoid:** The section in `index.astro` already has `<div class="relative z-10">` wrapping the section's content — import the components inside this div, not outside.

**Warning signs:** Cards appear very dim or completely invisible on sections that have a `tone-image` background.

### Pitfall 5: `font-display` Class vs. `font-family: var(--font-display)` in Tailwind v4

**What goes wrong:** Developer writes `class="font-display"` expecting Special Elite font, but the class doesn't work. In Tailwind v4, `@theme` token `--font-display` generates a utility class only if the token namespace matches.

**Why it happens:** The existing `global.css` defines `--font-display` in `@theme`. Tailwind v4 generates `font-display` as a utility class from `--font-display`. However, if there's a naming conflict with Tailwind's own `font-display` (related to `font-display` CSS property), behavior may be unexpected.

**How to avoid:** Verify `font-display` class works in existing `<h2>` headings. Per global.css base layer, `h1-h6` already get `font-family: var(--font-display)` automatically — no class needed on headings. For non-heading elements wanting Special Elite, use `style="font-family: var(--font-display)"` to be safe.

**Warning signs:** Cards show Space Mono instead of Special Elite on title text.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Client-side fetch for static data | Build-time `fs.readFileSync` in frontmatter | Astro 1.0+ (all versions) | Cards render as static HTML — no loading state, no JS required |
| JSON `import` from `src/` | `fs.readFileSync` from `public/` | Specific to this project's data pipeline | Avoids modifying generate-data.js; file stays in public/ for browser fetch |
| `__dirname` (CommonJS) | `process.cwd()` (ESM-compatible) | Node.js ESM adoption | `__dirname` is undefined in Astro ESM modules |

**No deprecated patterns:** This phase uses only stable, long-standing patterns.

## Open Questions

1. **Section layout: one section vs. two**
   - What we know: `#sectors` placeholder exists for "Gravel Sectors." `#info` placeholder exists for "Details, registration, GPX — Phase 7."
   - What's unclear: Should KOM and Restock go in `#sectors` alongside gravel sectors, or get their own visual grouping?
   - Recommendation: Put all three card types in `#sectors` since they're all route annotation data. Group them visually within the section using subheadings. `#info` remains for Phase 7.

2. **Card grid column count**
   - What we know: 6 sectors + 3 KOM + 4 restock = 13 items. Screen widths range from mobile to desktop.
   - What's unclear: Designer/owner preference for how many columns on desktop. Single-column vs. multi-column grid layout.
   - Recommendation: Use `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` for sectors; put KOM + restock in a narrower sidebar. Adjust once rendered visually.

3. **Tone image for `#sectors` section**
   - What we know: `#sectors` currently has no tone background image (unlike `#hero` and `#route`).
   - What's unclear: Whether the owner wants a tone image added to `#sectors`.
   - Recommendation: Default to no tone image — the `.classified-border` cards provide sufficient visual texture. Tone images are a design embellishment, not a requirement for this phase.
