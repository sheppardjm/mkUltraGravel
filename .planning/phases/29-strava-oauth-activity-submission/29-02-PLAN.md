Create `netlify/functions/strava-callback.js` using v1 handler syntax (`exports.handler`).

This function handles Strava's OAuth redirect back to the site. It:

1. **Handle access denied**: If `event.queryStringParameters.error === 'access_denied'`, redirect to `/submit?submit=denied`

2. **Verify CSRF state nonce**:
   - Parse `state` from query params: `Buffer.from(state, 'base64url').toString('utf8')` then `JSON.parse`
   - Extract `nonce` and `activityUrl` from decoded state
   - Parse cookies from `event.headers['cookie']` (split on `;`, then split on `=`, trim whitespace)
   - Compare `nonce` to `cookies['strava_oauth_state']`
   - If mismatch or missing: return 400 "Invalid or missing state parameter"

3. **Exchange authorization code for access token**:
   - POST to `https://www.strava.com/api/v3/oauth/token` with JSON body:
     - `client_id`: `process.env.STRAVA_CLIENT_ID`
     - `client_secret`: `process.env.STRAVA_CLIENT_SECRET`
     - `code`: from query params
     - `grant_type`: `'authorization_code'`
   - Parse response JSON. If no `access_token`, return 400 "Token exchange failed"

4. **Extract activity ID from activity URL**:
   - Match `/strava\.com\/activities\/(\d+)/` against `activityUrl`
   - If no match: return 400 "Invalid activity URL"

5. **Fetch activity with all segment efforts**:
   - GET `https://www.strava.com/api/v3/activities/${activityId}?include_all_efforts=true`
   - Authorization header: `Bearer ${tokenData.access_token}`
   - If response not ok: return 400 "Failed to fetch activity"

6. **Filter matching segment efforts**:
   - Define ALL_SEGMENT_IDS as a Set of all 9 event segment IDs (strings):
     `'24479270','24479292','41126651','24479426','24479467','24479496','34573011','16438243','6809754'`
   - Filter `activity.segment_efforts` where `ALL_SEGMENT_IDS.has(String(effort.segment.id))`
   - If zero matches: return 400 with HTML page explaining "No matching event segment efforts found in this activity. Make sure you submitted an activity from the MK Ultra Gravel course." Include a link back to /submit.

7. **Build result data for the confirmation page**:
   - Build segments object: `{ [String(effort.segment.id)]: { elapsed_time: effort.elapsed_time } }` for each matching effort
   - Build result payload: `{ athleteId: String(tokenData.athlete.id), name: (firstname + ' ' + lastname).trim(), activityUrl, segments }`
   - Base64url-encode the payload: `Buffer.from(JSON.stringify(payload)).toString('base64url')`

8. **Redirect to confirmation page**:
   - Return 302 to `/submit-confirm?data=${encodedPayload}`
   - Clear the CSRF cookie: `Set-Cookie: strava_oauth_state=; HttpOnly; Secure; SameSite=Lax; Max-Age=0; Path=/`

Important implementation notes:
- Use native `fetch` (no npm packages)
- Strava returns `segment.id` as a number -- always cast with `String()` before comparing to the Set
- The `tokenData` response includes `athlete.id`, `athlete.firstname`, `athlete.lastname` directly in the token exchange response (no separate API call needed for athlete info)
- Use `activity:read_all` was already requested in strava-auth.js; this function just uses the token
- Error states should return user-friendly HTML (not raw text) with a link back to /submit

**Part A: Create `src/pages/submit-confirm.astro`**

This is the confirmation page shown after successful OAuth callback. The rider sees their extracted data and must select their gender category and consent to public display before final submission.

Page structure:
1. Import and use `BaseLayout` with title "Confirm Your Submission -- MK Ultra Gravel"
2. In the Astro frontmatter, read `Astro.url.searchParams.get('data')` -- this is the base64url-encoded result payload from the callback
3. Decode the data: `JSON.parse(Buffer.from(data, 'base64url').toString('utf8'))` in frontmatter
4. If no data param or decode fails, show an error message with a link to /submit
5. Display extracted information:
   - Athlete name (from decoded data)
   - Number of matching segments found (e.g., "7 of 9 event segments matched")
   - Activity URL (as a clickable link to Strava)
6. A form that POSTs to `/api/submit-result` with:
   - Hidden input: `data` (the raw base64url string, passed through unchanged)
   - A `<select>` dropdown for gender (id="gender", name="gender", required):
     - Option value="" disabled selected: "Select your category"
     - Option value="M": "Men"
     - Option value="F": "Women"
     - Option value="NB": "Non-binary"
   - A checkbox (id="consent", name="consent", value="yes", required): "I consent to my name, category, segment times, and Strava activity link being displayed publicly on the MK Ultra Gravel results page."
   - Submit button: "Submit My Results"
7. Style consistently with submit.astro -- dark theme, centered container, display font headings, monospace labels

Important: The `data` hidden field passes through the base64url string exactly as received. The submit-result function will decode and validate it server-side.

**Part B: Create `netlify/functions/submit-result.js`**

This function handles the final form POST. It builds the per-athlete JSON, commits it to GitHub, and triggers a Netlify rebuild.

Use v1 handler syntax (`exports.handler`).

The function:

1. **Parse form data**: The body is `application/x-www-form-urlencoded`. Parse with `new URLSearchParams(event.body)` to get `data`, `gender`, and `consent`.

2. **Validate consent**: If `consent !== 'yes'`, return 400 "Consent is required."

3. **Validate gender**: If gender not in `['M', 'F', 'NB']`, return 400 "Invalid gender category."

4. **Decode and validate data**: Base64url-decode the `data` param, JSON.parse it. Verify it has `athleteId`, `name`, `activityUrl`, `segments`. If invalid, return 400.

5. **Build the athlete result object** conforming to Phase 28 schema:
   ```
   {
     athleteId: decoded.athleteId,
     name: decoded.name,
     gender: gender,           // from form, NOT from Strava profile
     activityUrl: decoded.activityUrl,
     submittedAt: new Date().toISOString(),
     segments: decoded.segments
   }
   ```

6. **Commit to GitHub via Contents API**:
   - File path: `public/data/results/athletes/${decoded.athleteId}.json`
   - First, GET the file to check if it exists (for SHA on update):
     - `GET https://api.github.com/repos/${GITHUB_OWNER}/${GITHUB_REPO}/contents/${path}`
     - Headers: `Authorization: Bearer ${GITHUB_TOKEN}`, `Accept: application/vnd.github+json`, `X-GitHub-Api-Version: 2022-11-28`
     - If 200: extract `sha` from response (needed for update)
     - If 404: file doesn't exist yet (first submission, no sha needed)
   - Then, PUT the file:
     - `PUT https://api.github.com/repos/${GITHUB_OWNER}/${GITHUB_REPO}/contents/${path}`
     - Body: `{ message: "result: ${name} (${gender})", content: Buffer.from(JSON.stringify(resultObj, null, 2) + '\n').toString('base64'), committer: { name: 'MK Ultra Gravel Bot', email: 'bot@mkultragravel.netlify.app' } }`
     - If sha exists, include `sha` in body
   - If PUT returns 409 (race condition): return a friendly HTML page saying "There was a conflict. Please try submitting again." with a link to /submit
   - If PUT returns other non-2xx: return 500 with error details

7. **Trigger Netlify rebuild**:
   - POST to `process.env.NETLIFY_BUILD_HOOK` with body `'{}'`
   - Fire-and-forget (don't fail the submission if rebuild trigger fails)

8. **Return success**: Redirect (302) to `/submit?submit=success` OR return an HTML success page directly. Preferred: return HTML directly with:
   - "Results Submitted!" heading
   - The athlete's name and category
   - "Your segment times have been submitted. The site will rebuild shortly with updated results."
   - A link to the home page and a link to /submit to submit another activity

Important implementation notes:
- All env vars read from `process.env`: GITHUB_TOKEN, GITHUB_OWNER, GITHUB_REPO, NETLIFY_BUILD_HOOK
- Do NOT put secrets in netlify.toml -- they must be in Netlify dashboard
- The `content` field in the GitHub API PUT must be base64-encoded (standard base64, NOT base64url)
- The commit message should identify the athlete: `result: FirstName LastName (M)`
- Use `additionalProperties: false` in the schema means only the 6 specified fields are allowed -- do not add extra fields
- Segment IDs in the segments object are strings (e.g., `"24479270"`, not `24479270`)
