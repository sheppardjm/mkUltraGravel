  <action>
Create scripts/assign-card-photos.js following the established pipeline script pattern (see generate-thumbnails.js for style reference). The script must:

1. Read public/data/photos.json (53 photos with `mi` field for mile marker)
2. Read public/data/annotations.json (has `sectors` array with startMi/lengthMi and `kom` array with startMi/lengthMi)
3. For each sector and KOM entry, select a coverPhoto using this two-pass algorithm:
   - Pass 1: Filter photos where `p.mi >= startMi && p.mi <= startMi + lengthMi`. If any found, pick the one closest to the midpoint of the segment.
   - Pass 2 (fallback): If no photos in range, pick the photo from the entire manifest closest to the midpoint. Log a warning: `WARNING: no photos within range {startMi}-{endMi}, using nearest fallback`
4. Set `annotation.coverPhoto = selectedFilename` on each sector and KOM object
5. Collect unique coverPhoto filenames (deduplicate -- same photo could theoretically match multiple annotations)
6. For each unique coverPhoto, generate a 600x338 WebP card crop:
   - Source: public/images/{filename}
   - Output: public/images/cards/{basename}.webp (strip original extension, add .webp)
   - sharp settings: `.resize(600, 338, { fit: 'cover', position: 'attention' }).webp({ quality: 80, effort: 4 })`
   - Skip if card file already exists (idempotent)
7. Write enriched annotations.json back with `JSON.stringify(annotations, null, 2) + '\n'`
   - CRITICAL: Do NOT strip or modify the existing `track`, `lat`, `lon`, `endLat`, `endLon`, `endMi` fields. Read the full object, mutate only coverPhoto, write back.
8. Log summary: card photos generated count, skipped count, annotations updated count

IMPORTANT: Do NOT sort the `photos` array in-place during fallback -- copy it first or use a non-mutating approach. The original array order must be preserved for subsequent iterations.

Export: `module.exports = { assignCardPhotos }` and `if (require.main === module)` pattern matching other scripts.
  </action>
