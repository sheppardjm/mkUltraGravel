<task type="auto">
  <name>Task 1: Run Lighthouse mobile audit on production build and address issues</name>
  <files>src/pages/index.astro</files>
  <action>
1. Build the production site:
   ```bash
   npm run build
   ```

2. Start the preview server (production build):
   ```bash
   npx astro preview
   ```
   Wait for it to be ready (typically port 4321).

3. Run Lighthouse in headless Chrome against the preview server using Playwright browser tools:
   - Navigate to `http://localhost:4321`
   - Use the Playwright MCP tools to evaluate page performance metrics

   Alternatively, run Lighthouse CLI:
   ```bash
   npx lighthouse http://localhost:4321 \
     --emulated-form-factor=mobile \
     --throttling-method=simulate \
     --output json \
     --output-path ./lighthouse-report.json \
     --chrome-flags="--headless=new"
   ```

4. Parse the Lighthouse report and check:
   - **LCP:** Must be < 2.5s (green). The hero WebP conversion from Plan 02 should have fixed this. If still > 2.5s, investigate which element is the LCP candidate and its resource load time.
   - **CLS:** Must be < 0.1 (green). Check if any images without explicit width/height cause layout shifts. The photo gallery thumbnails use `aspect-[3/4]` which should prevent CLS. The map container has a fixed height. Flag any unexpected shifts.
   - **INP (Interaction to Next Paint):** Must be < 200ms (green). The site has minimal JavaScript — Chart.js, Leaflet, PhotoSwipe all lazy-load. INP should be fine.
   - **Performance score:** Target 90+ on mobile.

5. If any metric fails:
   - **LCP still > 2.5s:** Check if other tone images (lazy-loaded) are interfering. Ensure no lazy images are in the initial viewport. Verify the WebP hero is actually being served (not the JPEG fallback).
   - **CLS > 0.1:** Add explicit `width` and `height` attributes to any images missing them. Check if the map or chart causes layout shift on load.
   - **INP > 200ms:** Profile which interaction is slow. Check for synchronous script execution blocking the main thread.

6. Apply fixes and re-run Lighthouse to confirm improvement.

7. Clean up: stop the preview server, delete lighthouse-report.json if created (or keep for reference — not committed to git).

**Important:** Lighthouse against `astro preview` (production build on localhost) is the correct measurement. Do NOT test against `astro dev` — dev server serves unoptimized assets.
  </action>

1. Lighthouse mobile Performance score >= 90
2. LCP < 2.5s on simulated mobile 4G
3. CLS < 0.1
4. `npm run build` succeeds (if any fixes were applied)
5. No critical Lighthouse warnings related to images, fonts, or scripts
  </verify>
